<objective>
Deliver the run-tracking I/O foundation of the v0.5.0 ETL layer: the sync `ETLAccessor`
class (in `etl.py`) holding `init()` / `_start_run` / `_end_run`, plus the lazy `db.etl`
property on `Database`. This phase mirrors the existing `spatial.py` / `db.spatial`
accessor pattern exactly (D-01/D-02). Run-log writes use a fresh short-lived autocommit
connection per write, fully independent of the load transaction (D-04/D-05), so a
`status='failed'` row commits even when the load txn rolls back. The `pipeline_runs` DDL
is the locked Phase 16 schema (D-09), consumed here, not re-authored.

Purpose: Every subsequent load phase (18/19) inherits correct transaction-boundary
behavior. The four ROADMAP success criteria are only literally testable once `db.etl`,
`init()`, the run-log primitives, and the auto-create seam exist.

Output:
- `pycopg/etl.py`: new `class ETLAccessor` (init, `_start_run`, `_end_run`, thin `run()` stub).
- `pycopg/database.py`: `self._etl` field + `etl` lazy property.
- `tests/test_etl_accessor.py`: unit + DB integration tests covering SC-1..SC-4.

NOT in this plan (Phase 18/19 scope): extract, transform execution, the three load modes,
`RunResult`, `history()`, `last_run()`, `dry_run`, `AsyncETLAccessor`. The `run()` body
here is a thin auto-create + start/end seam only — no extract/transform/load logic.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → ETLAccessor run-log writes | Pipeline name + row counts + error text flow into `pipeline_runs`. The pipeline name is caller-supplied; counts/timestamps are library-generated; error text derives from exception messages/tracebacks. |
| ETLAccessor → PostgreSQL (autocommit conn) | A dedicated short-lived autocommit connection, deliberately separate from the load transaction (the integrity-critical boundary of this phase). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | Tampering (SQL injection) | `_start_run` / `_end_run` / `init` SQL | mitigate | Run-log writes use the pre-authored `%s`-only constants (`ETL_INSERT_RUN`, `ETL_UPDATE_RUN`, `ETL_INIT_PIPELINE_RUNS`). NO identifier interpolation, NO f-strings, NO `schema` arg (D-08/D-12). `pipeline_runs` is a static unqualified literal in the constants. Therefore `validate_identifiers` is not needed for these writes (they touch no user identifiers) — and the pipeline name + error text are bound as `%s` values, never concatenated. SQL injection is not a live threat on this surface; this is stated explicitly per the threat-model requirement. |
| T-17-02 | Information disclosure | `error_message` / `error_traceback` persisted to `pipeline_runs` | accept (documented consideration) | Exception text and `traceback.format_exc()` could in principle contain secrets/credentials surfaced in an error (e.g. a connection-string fragment in a driver error). v0.5.0 records them verbatim for honest diagnostics (D-06).
