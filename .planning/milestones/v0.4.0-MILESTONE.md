# Milestone v0.4.0 — Quality & Spatial Helpers

**Statut :** proposé — en attente de validation
**Date de cadrage :** 2026-06-06
**Précédé par :** hotfix sécurité **v0.3.1** (release isolée, AVANT ce milestone)
**Source :** audit hors-GSD du 2026-06-06 ([.planning/AUDIT-2026-06-06.md](../AUDIT-2026-06-06.md)) + design Phase 8 spatial ([.planning/phases/08-spatial-helpers/08-DESIGN.md](../phases/08-spatial-helpers/08-DESIGN.md))

## 1. Intention

Faire de pycopg une lib **publiable proprement** : d'abord assainir (sécurité, parité, dette, tests, doc),
**puis** seulement ajouter les helpers spatiaux sur des fondations saines, et livrer (PyPI + ReadTheDocs).

Principe directeur de l'audit : **les abstractions existent déjà (`base.py`, `queries.py`) mais ne sont pas
branchées** → ~48 % de duplication. Ce milestone branche l'existant plutôt que de réécrire.

> Lève explicitement le *Out of Scope* historique de PROJECT.md :
> « Refactor Database into smaller classes/mixins — restructure in **future version** ». Cette version, c'est celle-ci.

## 2. Objectifs mesurables (Definition of Done du milestone)

| Axe | Cible | Mesure |
|-----|-------|--------|
| Sécurité | 0 injection SQL ; identifiants/intervalles validés partout | grep + tests d'injection dédiés au vert |
| Parité sync/async | 0 méthode divergente non documentée ; `test_parity` étendu (champs de retour) | test_parity étendu au vert |
| Dette/duplication | `base.py` & `queries.py` branchés ; duplication SQL ≈ 0 | revue + chute du % de duplication |
| Coverage | cliquet **70→80→90→95** (jamais redescendant) ; 100 % visé non-dur | `--cov-fail-under` monté par phase |
| Docstrings | **numpydoc**, peu profond, **sans exemples** ; couverture **interrogate ≥ 95 %** | `interrogate` en CI ≥ 95 |
| Spatial | `db.spatial.*` + `async_db.spatial.*` en parité (Phase 8) | tests builders + intégration PostGIS |
| Release | **v0.4.0** sur PyPI ; doc Sphinx à jour ; build RTD vert | wheel publié + RTD live |

### Conventions verrouillées
- **Docstrings** : format **numpydoc** (numpydoc, pas Google). Profondeur limitée (Summary + Parameters +
  Returns + Raises pertinents). **Pas de section Examples.** napoleon est déjà actif → activer `napoleon_numpy_docstring`.
- **Cliquet coverage** : gate globale `--cov-fail-under`, montée à chaque phase, **jamais abaissée** (cliquet).
  100 % = objectif méthode-par-méthode, pas une gate dure (s'arrête à 95 pour ne pas bloquer sur l'I/O dur à mocker).
- **interrogate** : ajouté en dep dev + config pyproject `fail-under = 95`, exécuté en CI.

## 3. Hors périmètre / pré-requis

- **v0.3.1 (hotfix sécurité)** est livré AVANT et n'est pas une phase de ce milestone (release séparée, faible risque).
- Pas de query builder / ORM / SQLAlchemy fluent (toujours Out of Scope de PROJECT.md).
- ETL complet = milestone ultérieur ; les helpers spatiaux en sont les briques mais on ne construit pas la couche ETL ici.
- Les 4 points ouverts de Phase 8 (`into=`, géométrie input, `unit=`, `where=`) **se tranchent au début de la
  phase spatiale** (via discuss-phase), pas dans ce document.

## 4. Phases proposées

Ordre = dépendances réelles : assainir les fondations → restaurer la parité → tests → doc → spatial → release.
Le refacto structurel (brancher base.py) vient APRÈS la parité, pour ne pas refactorer une cible mouvante.

### Phase 9 — Sécurité résiduelle & robustesse
**But :** fermer toute injection restante + bugs de correction.
- Valider identifiants/intervalles manquants : `add_compression_policy`/`add_retention_policy`, `create_spatial_index`,
  `vacuum`/`analyze`, `drop_index`, `to_dataframe`/`to_geodataframe`, `insert_many`/`upsert_many`, `valid_until`,
  `create_extension(schema)`, whitelist `grant`/`revoke`, options CSV.
- Bugs : `PooledDatabase.execute` commit manquant (B1), `session()` masque exception (B2), migrations sans
  transaction atomique (B3), `subprocess.os.environ` (B5).
- Tests d'injection dédiés (chaque correctif a son test rouge→vert).
- **Cliquet coverage → 80.**
- *(Note : si v0.3.1 a déjà absorbé une partie des fixes sécu, cette phase ne garde que le résiduel + les bugs.)*

### Phase 10 — Parité sync/async complète
**But :** restaurer la valeur cœur du projet.
- Implémenter dans AsyncDatabase : `add_primary_key`, `add_foreign_key`, `add_unique_constraint`,
  `truncate_table`, `drop_extension`, `database_exists`, `list_databases`, `create`/`create_from_env`.
- Implémenter dans Database : `insert_many`, `upsert_many`, `stream`, `notify`.
- Fixer C1 (`from_dataframe`/`from_geodataframe` async appliquent réellement `primary_key`),
  C2 (`AsyncDatabase.close()` dispose l'engine), C3 (driver `postgresql+psycopg_async`).
- Aligner signatures `create_extension(schema)`, `create_schema(owner)` et sémantique `table_info`/`list_roles`.
- **Étendre `test_parity.py`** : champs de retour + comportement mocké, pas seulement les noms.
- **Cliquet coverage → 90.**

### Phase 11 — Refactoring : brancher les abstractions
**But :** éliminer la duplication en utilisant ce qui existe déjà.
- **R1** brancher `queries.py` : remplacer les ~25 SQL inline par les constantes (source unique de vérité).
- **R3** adopter `base.py` : `Database(DatabaseBase, QueryMixin)` + idem async ; remonter
  `from_env`/`from_url`/`__repr__` ; utiliser `_build_batch_insert_sql`.
- **R4** extraire builders purs sans état : `_build_role_options`, `_build_pg_dump_cmd`, `_build_pg_restore_cmd`
  (testables sans DB → aide le coverage).
- Nettoyer le code mort résiduel (imports `re`, `stdout` non lu, `try/except: raise`, constantes `*_SIMPLE`).
- **Cliquet coverage → 95** (les builders purs sont faciles à couvrir).

### Phase 12 — Qualité documentaire (docstrings numpydoc + interrogate)
**But :** doc API homogène et mesurée.
- Migrer toutes les docstrings publiques au **format numpydoc**, peu profond, **sans Examples**.
- Ajouter `interrogate` (dev + config `fail-under=95`) + job CI.
- Activer `napoleon_numpy_docstring` dans Sphinx conf.
- Corriger le contrat d'exceptions (V2) : lever réellement `ExtensionNotAvailable`/`TableNotFound`/etc.
  au lieu de `RuntimeError`/`ValueError` (impacte les `Raises` des docstrings).
- Fixer `__version__` (V1) via `importlib.metadata`.
- Ajouter mypy en dev + config (TY1) ; annoter `async_engine` (TY2).
- *(Coverage : maintien 95, cliquet ne redescend pas.)*

### Phase 13 — Spatial helpers (Phase 8 réalisée)
**But :** `db.spatial.*` / `async_db.spatial.*`, sur fondations saines.
- **Début de phase : trancher les 4 points ouverts** (discuss-phase) → MAJ 08-DESIGN.md.
- Implémenter `pycopg/spatial.py` : builders SQL purs + `SpatialAccessor` / `AsyncSpatialAccessor`
  (réutilise le pattern builder-pur de Phase 11).
- ~10 helpers : contains, within, intersects, dwithin, distance, nearest, area, perimeter, centroid, buffer, transform.
- Garde PostGIS (`has_extension`), validation identifiants + valeurs `%s` (acquis de Phase 9).
- `test_parity` couvre l'accessor des 2 côtés. Tests builders (sans DB) + intégration PostGIS.
- *(Coverage : maintien 95.)*

### Phase 14 — Release v0.4.0 (PyPI + RTD)
**But :** livrer.
- MAJ doc Sphinx : remplacer les exemples `execute(...)` PostGIS par les helpers ; régénérer api-reference.
- Vérifier build RTD vert (`.readthedocs.yaml`).
- CHANGELOG v0.4.0 ; bump version cohérent partout ; MIGRATION notes si breaking.
- Publier wheel PyPI ; tag ; vérifier RTD live.
- Audit milestone (gsd-audit-milestone) avant archivage.

## 5. Dépendances entre phases

```
v0.3.1 (hotfix, hors milestone)
   │
   ▼
Phase 9 (sécu résiduelle) ─→ Phase 10 (parité) ─→ Phase 11 (refacto) ─→ Phase 12 (doc qualité)
                                                          │
                                                          ▼
                                              Phase 13 (spatial) ─→ Phase 14 (release)
```
Refacto (11) après parité (10) : inutile de factoriser des méthodes qui vont bouger.
Spatial (13) après refacto (11) : réutilise le pattern builder-pur. Doc (12) avant release mais peut chevaucher 13.

## 6. Risques

- **Migration docstrings massive** (numpydoc sur ~150 méthodes) → étaler, s'appuyer sur interrogate pour traquer.
- **Cliquet coverage** peut révéler du code dur à tester (subprocess, réseau) → les builders purs de Phase 11
  amortissent ; on plafonne la gate à 95.
- **Refacto base.py** touche les 2 gros modules → s'appuyer sur `test_parity` étendu (Phase 10) comme filet.
- **Breaking changes** (exceptions réellement levées, driver async) → documenter dans MIGRATION + bump mineur 0.4.0.

- Cliquet coverage **70→80→90→95** par phase, 100 % visé non-dur.
- 4 points Phase 8 tranchés **au début de la phase spatiale**.
- Livrable : doc milestone + ROADMAP, **validation utilisateur avant lancement GSD**.
