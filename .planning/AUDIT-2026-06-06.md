# Audit pycopg — 2026-06-06

**Type :** audit hors-GSD (qualité, sécurité, code mort, dette, refactoring)
**Méthode :** 4 agents Sonnet en parallèle + vérification manuelle des findings critiques à la source.
**Périmètre :** package `pycopg/*.py` (~6450 lignes), tests/, config.
**Statut findings critiques :** vérifiés dans le code source (non hallucinés).

---

## Verdict global

Architecture **bien pensée mais mal branchée** : les abstractions partagées (`base.py`, `queries.py`)
existent mais ne sont **pas utilisées** → ~48 % de duplication entre `database.py` (2376 l.) et
`async_database.py` (2249 l.). Trois axes bloquants avant une release PyPI stable :
1. **Sécurité** — 8+ injections SQL réelles (validations manquantes sur méthodes secondaires).
2. **Parité sync/async** — 10-12 méthodes divergentes (la valeur cœur du projet est entamée).
3. **Maturité release** — couverture tests 23,5 % (gate 70 % non tenue), `__version__` faux, 4 exceptions publiques jamais levées.

**Défense en profondeur (reco transverse) :** envisager `psycopg.sql.Identifier()` pour les fragments DDL
(validation + échappement natif) plutôt que la seule regex de `validate_identifier`.

## 2. PARITÉ SYNC/ASYNC (priorité 2 — valeur cœur)

### Bugs critiques de parité
- **C1 [CRITIQUE]** `from_dataframe` / `from_geodataframe` async : `primary_key` **silencieusement ignoré**
  (juste un `logger.warning` "wait for Phase 3"). La version sync l'applique. → contrat d'API mensonger.
  async:1465-1471 et 1587-1593.
- **C2 [HAUTE]** `AsyncDatabase.close()` = **no-op total** → `_async_engine` SQLAlchemy jamais disposé =
  fuite de ressources. async:2233. (Sync fait `_engine.dispose()`.)
- **C3 [HAUTE]** `async_engine` construit avec `postgresql+psycopg://` (driver **sync**) au lieu de
  `postgresql+psycopg_async://`. async_database.py:72 / config.py:210.

### Méthodes manquantes / divergentes
**Absentes d'AsyncDatabase :** `add_primary_key`, `add_foreign_key`, `add_unique_constraint`,
`truncate_table`, `drop_extension`, `database_exists`, `list_databases`, `create`/`create_from_env` (classmethods).
**Absentes de Database :** `insert_many`, `upsert_many`, `stream`, `notify` (et `listen` async-only — acceptable).
**Signatures divergentes :** `create_extension` (param `schema`), `create_schema` (param `owner`).
**Divergence sémantique non détectée par test_parity :** `table_info()` (8 vs 5 colonnes),
`list_roles()` (8 vs 5 champs).

### Angles morts de test_parity.py
Ne vérifie que l'existence des noms + noms de params. Ne détecte PAS : types de retour, **champs de données
retournés**, comportement à l'exécution (ignorer `owner`/`primary_key` silencieusement passe inaperçu).

## 3. BUGS (hors parité/sécurité)

- **B1 [HAUTE]** `PooledDatabase.execute` : `return cur.fetchall()` sans `commit()` préalable → les requêtes
  avec résultats (ex. `INSERT ... RETURNING`) sont rollback par le pool au retour de connexion. pool.py:152-158.
- **B2 [HAUTE]** `session()` sync : exception dans le `finally` (commit/close) masque l'exception originale. database.py:373-382.
- **B3 [HAUTE]** `migrations.py` `_apply`/`rollback` : pas de transaction explicite atomique (utiliser `db.transaction()`). migrations.py:232/287.
- **B4 [MOY]** `copy_to_csv` sync : curseur réutilisé après COPY pour le COUNT (état indéfini ; l'async fait mieux). database.py:2300-2308.
- **B5 [MOY]** `subprocess.os.environ` au lieu de `os.environ` (fragile). database.py:2135/2230/2249.
- **B6 [INFO]** `ConnectionError` custom masque le built-in Python → renommer `PycopgConnectionError`.

## 5. REFACTORING (axe demandé explicitement)

> La structure de refacto **existe déjà** dans `base.py`/`queries.py` — il s'agit de la **brancher**, pas de la réécrire.
> Contrainte absolue : préserver la parité sync/async. Levier : factoriser les **builders SQL purs sans état**
> (même code des 2 côtés) — exactement le pattern que le design Phase 8 (`db.spatial.*`) amorce.

### TOP refactorings (par impact / effort)

**R1 — Brancher `queries.py` partout [HAUTE, effort moyen, fort ROI]**
Remplacer les ~25 SQL inline dupliqués par les constantes existantes. Source unique de vérité ;
un fix se propage aux 2 classes. Le travail d'inventaire est déjà fait.

**R2 — Compléter la parité AsyncDatabase [HAUTE, effort moyen]**
10 méthodes manquantes (la plupart = adaptation triviale ~5 l.) + fix `close()` + URL `psycopg_async`.
Lève aussi C1/C2/C3.

**R3 — Adopter `base.py` comme couche partagée [HAUTE, effort élevé, structurant]**
Faire hériter `Database(DatabaseBase, QueryMixin)` et idem async ; remonter `from_env`/`from_url`/`__repr__`
(déjà écrits) ; utiliser `QueryMixin._build_batch_insert_sql`. Réduit ~400 l. de duplication.

**R4 — Extraire les builders purs sans état [MOY, effort moyen]**
`_build_role_options()`, `_build_pg_dump_cmd()`, `_build_pg_restore_cmd()` : identiques sync/async,
seul l'appel I/O diffère. Testables sans DB.

**R5 — Découpage modulaire par domaine [élevé, modulable]**
Sur le modèle Phase 8 : accessors `db.spatial` / `db.timescale` / `db.admin` + modules `_backup.py`.
`database.py`/`async_database.py` deviennent de fins coordinateurs. **Commencer par `_spatial` (déjà roadmappé Phase 8).**
