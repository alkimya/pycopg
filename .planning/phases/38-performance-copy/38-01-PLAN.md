<objective>
Route `from_dataframe` data load through the PostgreSQL COPY protocol (psycopg v3) instead of `df.to_sql(con=engine)`, for BOTH `Database` and `AsyncDatabase`, while preserving the observable contract (`if_exists`, `index`, `primary_key`, and `dtype` "for free"). Introduce the private COPY-streaming helpers consumed by this plan and by the ETL seam in Plan 02.

This plan implements the Hybrid DDL+COPY architecture locked in CONTEXT.md: `df.head(0).to_sql(...)` creates/replaces the typed empty table (delegating type inference + `if_exists` + `index` semantics to pandas/SQLAlchemy), then rows stream via COPY on a separate psycopg connection.

Purpose: Eliminate the slow `df.to_sql` row-by-row insert path on the most-used bulk-load API (PERF-01), and ship the shared COPY helper that Plan 02 (ETL) reuses. Maintain sync/async parity (PERF-05).
Output: `_stream_df_copy` + `_async_stream_df_copy` module-level helpers; rewritten `from_dataframe` (sync + async); new real-DB COPY-path tests; dedicated async COPY behavioral test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → from_dataframe (table/schema/column identifiers) | Identifiers are interpolated into the `COPY {schema}.{table} ({cols}) FROM STDIN` SQL string — untrusted-identifier surface |
| caller → from_dataframe (DataFrame cell values) | Row VALUES cross into the COPY data stream via `write_row` — parameterized by the COPY protocol, never string-formatted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-01 | Tampering (SQL injection via identifier) | `_stream_df_copy` / `_async_stream_df_copy` COPY SQL string (`COPY {schema}.{table} ({cols_str}) FROM STDIN`) | mitigate | Caller (`from_dataframe`) MUST call `validate_identifiers(table, schema)` as its FIRST statement before the helper is reached — exactly as `copy_insert` does today (database.py L1028). The helper never builds SQL from un-validated identifiers; `columns` derive from `df_ddl.columns` which were already used to create the table via the typed DDL. Verified by the Task 2 source assertion that `validate_identifiers` is first. |
| T-38-02 | Tampering (SQL injection via value) | DataFrame cell values streamed through `write_row` | accept (already mitigated by design) | User values go through `copy.write_row(...)` which the psycopg COPY protocol encodes/escapes per registered type adapter — values are NEVER string-interpolated into the COPY SQL. The NaN/NaT→None normalization substitutes Python `None` (→ SQL NULL), not a string. No new value-interpolation path is introduced. |
| T-38-03 | Information disclosure (replace leaves empty table on COPY failure) | `from_dataframe(if_exists="replace")` two-phase (D-04) | accept (documented) | DDL commits on the engine before COPY; a COPY failure leaves the table replaced-but-empty. This is the pre-existing `replace` semantics ("drop and rebuild") and is now DOCUMENTED in the docstring (D-04). Atomic DDL+COPY deferred to v1.0.0. |
| T-38-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed this phase (zero new runtime deps, RESEARCH "Package Legitimacy Audit" = N/A). No install task exists, so no legitimacy checkpoint required. |
</threat_model>
