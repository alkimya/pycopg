<objective>
Route the ETL load path for `append` and `replace` through the COPY protocol, streaming rows inline on the transaction cursor (the seam) using the `_stream_df_copy` / `_async_stream_df_copy` helpers shipped in Plan 01 — eliminating the `df.astype(object).where(pd.notnull(df), None).to_dict(orient="records")` full-frame materialization for these modes. `upsert` stays on `INSERT ... ON CONFLICT` (COPY cannot do ON CONFLICT), keeping its `list[dict]` materialization.

Purpose: Remove the heavy full-frame copy + dict materialization from the two volume load modes (PERF-02), while keeping `db.etl.run()`'s observable contract (status, rows_loaded, parity) identical and preserving the atomic run-log/load seam (D-02a). Maintain sync/async parity (PERF-05).
Output: rewritten sync + async ETL load steps 3-6; module-level import of the Plan 01 helpers into etl.py; ETL COPY-path tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Pipeline config → ETL seam (target/schema/columns) | `pipeline.target` / `pipeline.schema` / `columns` are interpolated into the COPY SQL inside the helper — identifier surface |
| extracted/transformed DataFrame cell values → COPY stream | Row VALUES cross into the COPY data stream via `write_row` — parameterized by the COPY protocol, never string-formatted |
| run-log writes ↔ load (atomicity seam) | The run-log isolation seam must remain atomic — load and run-log status must not diverge |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-04 | Tampering (SQL injection via identifier) | ETL COPY SQL built from `pipeline.target` / `pipeline.schema` / `columns` inside `_stream_df_copy` | mitigate | The ETL pipeline path validates identifiers upstream (existing builders `_build_insert_sql`/`_build_upsert_sql`/`build_truncate_sql` and the `from_dataframe` head(0) DDL all run `validate_identifiers` first). `columns` derive from `df.columns` of the same DataFrame that created/targets the table. No NEW un-validated identifier-interpolation path is introduced by routing through the COPY helper — the helper does not invent identifiers, it receives them from the already-validated pipeline flow. |
| T-38-05 | Tampering (SQL injection via value) | DataFrame cell values streamed through `write_row` on the transaction cursor | accept (mitigated by design) | Values go through `copy.write_row(...)` (COPY protocol-encoded), never string-interpolated. NaN/NaT→None substitutes Python `None`. Identical value-safety to the prior INSERT-param path. |
| T-38-06 | Repudiation / atomicity break (run-log vs load divergence) | ETL load seam — risk of calling a method that opens its own connection+commit | mitigate | D-02a: COPY runs INLINE on the transaction cursor; the public `copy_insert` (which opens its own connection + commits) is NEVER called in the seam. Task 2 source assertion proves `copy_insert` does not appear in the seam region. A COPY failure rolls back with the transaction, keeping run-log status consistent with the load (run-log marked `failed` on exception). |
| T-38-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages installed (zero new runtime deps). No install task, no legitimacy checkpoint required. |
</threat_model>
