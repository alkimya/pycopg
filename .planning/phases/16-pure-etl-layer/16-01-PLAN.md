  <action>
    Append a new `# ETL QUERIES` banner section (same 79-char `=` rule style) at the
    end of queries.py, then define exactly these five SCREAMING_SNAKE_CASE
    triple-quoted constants. Use `%s` placeholders for all values; NEVER f-string /
    `format()` an identifier into a constant (D-12).

    1. `ETL_INIT_PIPELINE_RUNS` — idempotent `CREATE TABLE IF NOT EXISTS pipeline_runs`
       (D-14/D-15). Columns (reconciled with Phase 17 SC-1 per CONTEXT "Claude's
       Discretion"): `run_id BIGSERIAL PRIMARY KEY`, `pipeline_name TEXT NOT NULL`,
       `started_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `finished_at TIMESTAMPTZ`,
       `status TEXT NOT NULL CHECK (status IN ('running','success','failed'))`
       (TEXT+CHECK, NOT a PG ENUM — D-14), `rows_extracted BIGINT`,
       `rows_loaded BIGINT`, `error_message TEXT`, `error_traceback TEXT`,
       `watermark JSONB` (nullable, always NULL in v0.5.0 — D-14). No `%s` needed
       (static DDL).
    2. `ETL_INSERT_RUN` — INSERT a new run row at start. `%s` placeholders for
       `pipeline_name`, `status`, `started_at` (and any other start-time columns);
       `RETURNING run_id`. Builder lands in Phase 17 (D-10); author the constant now.
    3. `ETL_UPDATE_RUN` — UPDATE the run row at finish: set `status`, `finished_at`,
       `rows_extracted`, `rows_loaded`, `error_message`, `error_traceback` via `%s`,
       `WHERE run_id = %s`. Builder lands in Phase 17 (D-10).
    4. `ETL_LIST_RUNS` — SELECT run rows for one pipeline, newest-first
       (`WHERE pipeline_name = %s ORDER BY started_at DESC LIMIT %s`). Builder lands
       in Phase 19 (D-10).
    5. `ETL_GET_LAST_RUN` — SELECT the single most-recent run for one pipeline
       (`WHERE pipeline_name = %s ORDER BY started_at DESC LIMIT 1`). Builder lands
       in Phase 19 (D-10).

    The status CHECK value set is exactly `'running'`, `'success'`, `'failed'`. Do
    NOT use a PG ENUM, `parent_run_id`, `trigger_source`, `depends_on`, or any
    DAG/scheduler column (Pitfall 10/11). If any constant ends up containing a
    literal `%` (e.g. a LIKE pattern), double it to `%%` — none of these five are
    expected to need it.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Pipeline definition → generated SQL | The only crossing in the pure layer: caller-supplied identifiers (table/schema/conflict columns) would reach SQL text. No values flow in this phase (constants are static; no builder interpolates a value here). |

This phase has no I/O, no network, no auth, no user session. The only relevant
threat surface is SQL **identifier** injection. Value injection is N/A in this plan:
the five constants are static SQL with `%s` placeholders only — no identifier and no
value is interpolated.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | `queries.py` ETL constants | mitigate | All five constants use `%s` placeholders; NO f-string/`format()` identifier interpolation (D-12). Status values are a fixed `CHECK` literal set, not interpolated. |
| T-16-02 | Elevation of Privilege | `pipeline_runs` DDL evolution | accept | `status TEXT ... CHECK` instead of a PG ENUM (D-14) — avoids non-transactional `ALTER TYPE` in v0.6.0. Schema additivity is a design choice, not a runtime threat; accepted by design. |

Block-on severity: high. No `[ASSUMED]`/`[SUS]`/`[SLOP]` packages — this phase adds
zero dependencies (stdlib + existing pycopg only), so no `T-16-SC` supply-chain
checkpoint is required.
