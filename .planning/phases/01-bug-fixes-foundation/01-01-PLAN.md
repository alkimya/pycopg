  <files>pycopg/database.py</files>
  <action>
Fix two bugs in `pycopg/database.py`:

**BUG-01: Session cleanup guarantee (lines 340-353)**

The `session()` context manager currently has this finally block:
```python
finally:
    if not autocommit:
        self._session_conn.commit()
    self._session_conn.close()
    self._session_conn = None  # NEVER EXECUTES if close() raises
```

Replace with nested try/finally pattern:
```python
finally:
    try:
        if not autocommit:
            self._session_conn.commit()
        self._session_conn.close()
    except Exception:
        # Cleanup failure - connection may already be closed or broken.
        # Don't suppress: let the exception propagate after state reset.
        raise
    finally:
        self._session_conn = None  # ALWAYS executes
```

Note: No logging import needed here — the exception itself propagates and provides the error context. The critical fix is that `_session_conn = None` ALWAYS executes via the inner finally, even if commit() or close() raises.

**BUG-02: Transaction detection (line 281)**

The `cursor()` context manager currently only checks IDLE:
```python
if not autocommit and self._session_conn.info.transaction_status == TransactionStatus.IDLE:
    self._session_conn.commit()
```

Replace with complete TransactionStatus handling:
```python
if not autocommit:
    status = self._session_conn.info.transaction_status
    if status == TransactionStatus.INTRANS:
        self._session_conn.commit()
    elif status == TransactionStatus.INERROR:
        self._session_conn.rollback()
    # IDLE: no open transaction, nothing to do
    # ACTIVE: should not occur at cursor exit (mid-query)
    # UNKNOWN: connection in bad state, skip
```

Key behavior changes:
- INTRANS (valid open transaction) now triggers commit — previously was silently skipped
- INERROR (failed transaction) now triggers rollback — previously was silently skipped, which would cause subsequent queries to fail
- IDLE no longer triggers a spurious commit() call
