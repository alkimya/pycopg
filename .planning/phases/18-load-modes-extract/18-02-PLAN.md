<objective>
Replace the Phase 17 `run()` stub (`pycopg/etl.py:425-448`) with the real
extract â†’ transform â†’ load body, wired between the existing autocommit run-log writes.
Change the signature to `run(self, pipeline: Pipeline) -> int`. The load dispatches on
`pipeline.load_mode` and executes the Plan 01 builders' `(sql, params)` **directly on the
connection yielded by `db.transaction()`** (opened inside an internal `db.session()`), making
replace's TRUNCATE+INSERT atomic (SC-3). On any failure, record a failed run on the dedicated
autocommit connection and re-raise (OD-2 / domain exceptions).

Purpose: This is the architectural crux of Phase 18 â€” the corrected atomicity seam (RESEARCH
Q1) is the only way to satisfy SC-3 with the current `Database` API. Getting the connection
lifetimes right (load SHARES one transactional conn; run-log STAYS isolated on autocommit
conns) is exactly the Phase 17/18 duality and the highest-risk part of the milestone.
Output: A working `run(pipeline)` body in `etl.py` returning the bare `run_id` (int).
</objective>

<decisions_implemented>
- D-01: all row writes go through psycopg-native primitives inside one `db.transaction()` (never
  `from_dataframe`/`to_sql` for the row insert â€” that runs on the SQLAlchemy engine, a different
  connection). append â†’ insert; replace â†’ TRUNCATE + insert in one txn; upsert â†’ INSERT â€¦ ON CONFLICT.
- D-02 (as corrected by RESEARCH Q1): the atomic seam runs the load inside an internal `db.session()`
  and executes the load SQL directly on the `db.transaction()`-yielded conn â€” NOT via the public
  `insert_batch`/`upsert_many` (which commit at cursor exit / crash inside a txn block). Run-log
  writes STAY isolated on the dedicated autocommit conn (opposite direction â€” ETL-09 non-regression).
- D-03 / D-03a: target existence checked via `table_exists` before the load; append/upsert missing â†’
  `ETLTargetNotFoundError`; replace missing â†’ zero-row `from_dataframe(if_exists='replace')` create
  BEFORE opening the txn (no hand-built CREATE TABLE).
- D-05: transform dispatch â€” None no-op, single callable applied once, list applied in sequence.
- D-06: transform exception â†’ `ETLTransformError` naming the failing step (1-based index + label via
  `_step_label`), chained `from exc`; failed run recorded via `_end_run(status='failed', â€¦)`.
- D-07: post-transform DataFrame â†’ `list[dict]` with NaN/NaT coerced to None
  (`df.astype(object).where(pd.notnull(df), None).to_dict('records')`); `rows_loaded` = sum of
  `cur.rowcount`; tz-localization left to the caller and documented (matches `from_dataframe`).
- OD-2: on failure, record the failed run then re-raise the original/domain exception (no wrapper).
</decisions_implemented>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller â†’ `Pipeline.source` (SQL string) â†’ extract query | `source` is the caller's own SQL by design (same-DB, trusted author); wrapped as a subquery for LIMIT. |
| caller â†’ `Pipeline.target`/`schema`/`conflict_columns` â†’ load SQL | Identifiers cross into SQL construction in the load builders. |
| caller â†’ `Pipeline.extract_limit` â†’ LIMIT clause | Integer limit crosses into the extract query. |
| load txn conn â†” run-log autocommit conn | Two deliberately separate connection lifetimes (ETL-09). |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-03 | Tampering | `extract_limit` in the LIMIT clause | mitigate | Bound as a `:lim` param via `to_dataframe(params={"lim": n})`, never f-string-interpolated; already validated as a positive int at `Pipeline` construction (etl.py:186-197). |
| T-18-04 | Tampering | table-source name in the extract `SELECT *` | mitigate | `validate_identifiers(source, schema)` before building the table ref (or `to_dataframe(table=...)` which validates internally). |
