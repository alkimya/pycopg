<objective>
Add the READ helpers to BOTH `Database` and `AsyncDatabase`: `exists` (CRUD-04), `count` (CRUD-05),
`paginate` (CRUD-06), and `fetch_all` (CRUD-07, dict-fetch), with full sync/async parity (CRUD-08,
auto-enforced by `tests/test_parity.py`). `exists`/`count` build their optional WHERE via the
`_build_where_dict` builder from 34-01 (they accept `where=None`; the D-04 destructive guard does NOT
apply to reads). `paginate` extends the SELECT scaffold with a validated `order_by` column list, a
whole-clause `descending: bool`, int-cast `limit`/`offset`, and an optional dict-WHERE.
`fetch_all` is the thin `list[dict]` discoverability twin to `fetch_one` (D-08) — NO `into=` toggle,
NO tuples path. This plan also documents the dicts-by-default behavior (CRUD-07).

Purpose: Ergonomic existence/count/paged reads and an explicit dict-returning fetch, all
injection-safe (validated identifiers, `%s`-bound values).
Output: Four new public methods on each class + live-DB tests (sync + async) + a docstring note that
the core returns dicts by default. Parity is auto-covered once the methods exist on both classes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → exists/count/paginate | table/schema/column identifiers (incl. order_by columns and where keys) + where values from application code |
| caller → fetch_all | raw SQL string supplied by application code (caller-owned, like `execute`/`fetch_one`) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-06 | Tampering (SQL injection via identifier) | table/schema, where keys, order_by columns in exists/count/paginate | mitigate | `validate_identifiers` on table, schema, where keys (via `_build_where_dict`) and order_by columns before interpolation; order_by takes COLUMN NAMES not raw SQL (D-10) |
| T-34-07 | Tampering (SQL injection via value) | where values, limit/offset in exists/count/paginate | mitigate | where values bound as `%s`; limit/offset `int()`-cast before interpolation (base.py:203-206 idiom) |
| T-34-08 | Tampering (SQL injection via raw SQL) | `fetch_all(sql, params)` | accept | `fetch_all` is a thin twin to existing `fetch_one`/`execute` — caller owns the SQL string and must parameterize via `params`; same trust model as the pre-existing fetch helpers (no new surface) |
| T-34-SC | Tampering | npm/pip/cargo installs | accept | no new runtime dependencies (milestone constraint); nothing installed |
</threat_model>
