<objective>
Add the predicate-driven and single-row WRITE helpers to BOTH `Database` and `AsyncDatabase`:
`upsert` (CRUD-01), `delete_where` (CRUD-02), `update_where` (CRUD-03), with full sync/async parity
(CRUD-08, auto-enforced by `tests/test_parity.py`). `upsert` is the singular `RETURNING *` complement
to `upsert_many` (returns `dict | None`); `delete_where`/`update_where` build their WHERE via the
`_build_where_dict` builder from plan 34-01, return `cur.rowcount` (`int`), and raise `ValueError` on
empty/None `where` BEFORE any DB round-trip (D-04 destructive guard).

Purpose: Ergonomic single-row and predicate writes that stay injection-safe (validated identifiers,
`%s`-bound values) and never silently emit a WHERE-less DELETE/UPDATE.
Output: Three new public methods on each class + their live-DB tests (sync + async). Parity is covered
automatically once the methods exist on both classes — no per-method parity entry is added.
</objective>

  <behavior>
    - `db.upsert("t", {"id": 1, "name": "a"}, ["id"])` on a fresh row inserts and returns the full row
      dict (via `RETURNING *`), e.g. `{"id": 1, "name": "a", ...}`.
    - Calling again with `{"id": 1, "name": "b"}, ["id"]` updates name and returns the updated row dict.
    - Return type is `dict | None`. NOTE: because this builds `... ON CONFLICT (...) DO UPDATE SET ... RETURNING *`,
      every call affects exactly one row, so the runtime return is STRUCTURALLY always a `dict`. The `None`
      arm is DEFENSIVE-ONLY (it covers `cur.fetchone()` returning `None`, e.g. a future `DO NOTHING`/no-row
      variant); it is NOT a reachable branch of the current `DO UPDATE` SQL and is therefore not asserted by
      a behavioral test. Do NOT add a `DO NOTHING` path to force the `None` branch in this phase.
    - `update_columns` omitted → defaults to all non-conflict columns (same rule as `upsert_many`).
    - The async `AsyncDatabase.upsert` has the identical signature and returns `dict | None`.
  </behavior>
  <action>
    Add `upsert(self, table, row: dict, conflict_columns: list[str], update_columns: list[str] | None = None,
    schema: str = "public") -> dict | None` to `Database` (place next to `upsert_many`), and the
    `async def` twin to `AsyncDatabase`. Signature mirrors `upsert_many` (D-02: `conflict_columns`
    required positional; `update_columns` optional). Build the SQL (D-01/D-03): derive
    `columns = list(row.keys())`; default `update_columns` to non-conflict columns; call
    `validate_identifiers(*conflict_columns)` and `validate_identifiers(*update_columns)` (reuse the
    upsert_many shape at database.py:633-645); compose
    `INSERT INTO {schema}.{table} (...) VALUES (...) ON CONFLICT ({conflict_str}) DO UPDATE SET {update_str} RETURNING *`
    using `_build_insert_sql(table, columns, schema, on_conflict=...)` for the VALUES/identifier scaffold
    and appending ` RETURNING *` to the returned sql; params are `[row[c] for c in columns]`. Execute and
    return the single row: sync `with self.cursor() as cur: cur.execute(sql, params); return cur.fetchone()`;
    async `async with self.cursor() as cur: await cur.execute(sql, params); return await cur.fetchone()`.
    `cur.fetchone()` returns the affected row as a dict; the declared `-> dict | None` keeps `None` as a
    DEFENSIVE arm for a `cur.fetchone()` that yields nothing (not reachable under the current `DO UPDATE`
    SQL — every call affects one row — so do NOT engineer a `DO NOTHING` path to exercise it in this phase).
    NEVER interpolate a row value — all values bind via `%s`. Numpydoc shallow docstring (no Examples).
  </action>
