<objective>
Introduce the shared pure builder `_build_where_dict(where: dict) -> tuple[str, list]` on
`QueryMixin` in `pycopg/base.py` (D-12). It converts a dict of equality conditions
(`{col: val, ...}`) into an AND-ed `col = %s` SQL fragment plus a positional params list, with
every column key validated through `validate_identifiers` before interpolation. This is the
safety foundation reused by `delete_where`, `update_where`, `exists`, `count`, and `paginate` in
later plans. It is SEPARATE from the legacy `_build_select_sql(where: str)` raw-string path — do
NOT overload that param (D-12).

Purpose: Centralize the validated dict→`%s` predicate construction so every predicate-driven CRUD
helper binds user values as parameters and never interpolates a raw value or unvalidated identifier.
Output: One new pure staticmethod in base.py and its unit tests in test_base.py.
</objective>

    - CONTRACT: the builder is ONLY called with a non-empty dict. It does NOT guard against `{}` — callers
      (`delete_where`/`update_where` via the D-04 ValueError guard; `exists` via its own ValueError guard;
      `count`/`paginate` by routing AROUND the builder when `where is None`) guarantee a non-empty dict
      reaches it. (`_build_where_dict({})` would return `("", [])` and produce a malformed `WHERE ` — never
      let an empty dict reach the builder.)
  </behavior>
  <action>
    Add a pure `@staticmethod _build_where_dict(where: dict) -> tuple[str, list]` to the `QueryMixin`
    class in pycopg/base.py, placed next to `_build_select_sql`. Behavior:
    (1) call `validate_identifiers(*where.keys())` FIRST (mirrors base.py:104, base.py:192-194);
    (2) build the fragment as `" AND ".join(f"{col} = %s" for col in where)`;
    (3) build params as `list(where.values())`;
    (4) return `(fragment, params)`.
    The fragment is meant to be interpolated as `WHERE {fragment}` by callers, with params bound
    positionally — mirroring how `_build_select_sql` (base.py:199-200) appends `WHERE {where}`, but
    safe because values are `%s` placeholders, not interpolated. Do NOT add any DB access. Do NOT
    handle/guard an empty dict here — the builder is a MINIMAL primitive that assumes a non-empty dict.
    Empty/None rejection is the CALLER's job: `delete_where`/`update_where` raise `ValueError` on empty
    where (D-04); `exists` raises `ValueError` on empty/None where (no meaningful existence check without
    a predicate); `count`/`paginate` accept `where=None` and route AROUND this builder (omit the WHERE
    clause entirely) rather than passing `{}`. Calling `_build_where_dict({})` would yield `("", [])` →
    a malformed `WHERE ` — that path must be unreachable. Do NOT modify `_build_select_sql`'s existing
    `where: str` param.
    Add a numpydoc shallow docstring (Parameters/Returns; no Examples — project convention).
  </action>
