<critical_findings>
PLANNER LIVE-VERIFICATION (2026-06-23, local TimescaleDB 2.28.0, `timescaledb.license = apache`):

1. `time_bucket(%s, ts)` runs FOR REAL on the local Apache build — it is Apache-free.
   It returns a deterministic bucketed result; the `WHERE <col> >= %s AND <col> < %s`
   range with `datetime` binds also works. D-08 holds FOR `time_bucket` only.

2. **D-08 IS MATERIALLY WRONG FOR `time_bucket_gapfill`.** On the local Apache 2.28.0
   build, BOTH `time_bucket_gapfill(%s, ts)` and `time_bucket_gapfill(%s, ts, %s, %s)`
   raise `psycopg.errors.FeatureNotSupported`:
   `function "time_bucket_gapfill" is not supported under the current "apache" license`.
   `locf()` / `interpolate()` are reachable only inside a gapfill query, so they are
   gated the same way. This is the SAME license gate as Phase-31 continuous aggregates,
   NOT the Apache-free behavior D-08 assumed. The SQL *template* (4-arg gapfill +
   double-bound WHERE, `>=`/`<`) is otherwise syntactically valid — the error is a
   license gate, not a syntax error.

Consequence for plan 02 (live tests): `time_bucket` live tests assert REAL output;
`time_bucket_gapfill` live tests MUST tolerate `FeatureNotSupported` via try/except
(the Phase-31 cagg pattern at tests/test_timescale.py:1395-1419), with the mock
SQL-shape test as the authoritative assertion. This does NOT change the production
code in this plan — the builders/methods are identical regardless of license.
</critical_findings>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → query helper | `table` / `time_column` / `schema` identifiers cross here; `bucket_width` / `start` / `finish` values cross here |
| caller → structural SQL | `aggregates` and `where` are caller-supplied SQL FRAGMENTS, documented as structural SQL (not untrusted end-user input), same posture as the existing spatial accessor |
| query helper → PostgreSQL/TimescaleDB | read-only `SELECT`; no DDL, no autocommit seam, no writes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Tampering (SQL injection via identifiers) | `_build_time_bucket_sql` / `_build_time_bucket_gapfill_sql` `table`/`time_column`/`schema` | mitigate | `validate_identifiers(table, schema, time_column)` called at the top of each builder before any interpolation (utils.py:107) |
| T-32-02 | Tampering (value injection) | `bucket_width`, `start`, `finish`, where-fragment values | mitigate | all runtime values bound as `%s` positional params (psycopg `execute`) / `:pN` named binds (`to_dataframe` path via `_to_named_binds`); never string-interpolated |
| T-32-03 | Tampering (structural-SQL fragment) | `aggregates`, `where` args | accept | documented as caller-supplied STRUCTURAL SQL (not untrusted end-user input); identical trust posture to the shipped spatial accessor `where=`/expression args; no new attack surface for the lib's documented contract |
| T-32-04 | Denial of Service | unbounded `time_bucket` scan (no required range) | accept | read-only query under the caller's own connection/statement-timeout; same posture as any user SELECT; out of scope for a query helper |
| T-32-SC | Tampering | npm/pip/cargo installs | n/a | zero new runtime dependencies in this phase (D-12); no package installs, no audit table required |
