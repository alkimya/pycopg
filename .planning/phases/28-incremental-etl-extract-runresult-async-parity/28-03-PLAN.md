    Add a `## Incremental loading` section to `docs/etl.md` (D-A4) containing, in
    prose + a worked fenced `python` example (style-matched to the existing
    `## Defining a Pipeline` examples):

    1. A worked example: `Pipeline(name=..., source=..., target=...,
       incremental_column="updated_at", load_mode="upsert",
       conflict_columns=[...])` followed by `db.etl.run(p)` (and a note that
       `async_db.etl.run(p)` behaves identically — Core Value parity).
    2. Watermark-column requirements: must be monotonic / non-decreasing, an
       aware datetime (offset preserved, NOT coerced to UTC) OR an integer OR
       text; single-column only; the boundary is `>` (exclusive); float columns
       are rejected with an `ETLError`.
    3. Why `upsert` is required: `incremental_column` with `load_mode` in
       {append, replace} is forbidden at construction (`ValueError`); upsert makes
       re-loading the boundary row idempotent. Reference the construction guard.
    4. First-run / subsequent-run semantics: first run (no prior successful
       watermark) is a full load that records `max(incremental_column)`;
       subsequent runs extract only `col > last_watermark`. The watermark is read
       from the last SUCCESSFUL run and advanced only on success; a failed run
       does not advance it; an empty batch preserves the prior watermark (never
       writes NULL).
    5. The two new `RunResult` fields: `watermark_used` (the filter floor applied
       this run; `None` on the first run and for non-incremental pipelines) and
       `watermark_recorded` (the new high-water mark persisted; `None` for
       non-incremental and for empty/all-NULL batches). Note `history()` /
       `last_run()` surface `watermark_recorded` from stored rows with
       `watermark_used` always `None` (it is a per-run input, never stored).
    6. `dry_run` preview behavior: an incremental `dry_run=True` reads the prior
       watermark, applies the SAME filter, reports `watermark_used` +
       `watermark_recorded` (max of the filtered batch), and writes NO
       `pipeline_runs` row — so `rows_extracted` is an honest "what would a real
       run pull" number.
    7. Backfill / reset workflow (D-A5): there is no reset API; to force a full
       reload, neutralize the last successful watermark directly with manual SQL,
       e.g. `UPDATE pipeline_runs SET watermark = NULL WHERE pipeline_name = %s`
       (or DELETE the run rows); the next `run()` reads `None` → full extract →
       records a fresh watermark. Note that an `initial_watermark` first-run bound
       is coming in v0.8.0.

    Keep Sphinx `-W`-clean: no broken cross-references, no malformed directives,
    valid fenced `python` blocks. Do NOT bump any version or touch CHANGELOG /
    MIGRATION (Phase 29). Do NOT document a `reset_watermark()` API (rejected — D-A5).
