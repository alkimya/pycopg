<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo/CI → PyPI | The only outward-facing action: publishing the package to a public index. A compromised credential or unintended publish is the primary threat. |
| live PyPI → user machine | The clean-venv smoke installs from the public index; install-time code execution is the trust boundary being validated. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-06 | Spoofing/Elevation | PyPI publish credentials | mitigate | OIDC trusted publishing via publish.yml (id-token: write, environment pypi) — NO long-lived API token exists to leak; PyPI verifies the workflow identity. |
| T-33-07 | Tampering | unintended/automated publish | mitigate | Release is human-gated (D-03, autonomous:false): the publish only fires on a human-created GitHub Release `published` event (Task 3 checkpoint). No automated push to PyPI. |
| T-33-08 | Tampering (build integrity) | built sdist/wheel | mitigate | `uv lock --check` + `uv build` run locally (Task 2) AND again in publish.yml before upload, so the published artifact is built from the locked, committed release commit. |
| T-33-09 | Repudiation | release provenance | accept | Tag + GitHub Release + workflow run + RELEASE-LOG.md provide an auditable trail; no additional signing introduced this phase (consistent with v0.5–0.7 baseline). |
| T-33-SC | Tampering | clean-venv install from PyPI | mitigate | The smoke installs the exact pinned `pycopg==0.8.0` from the official index and only imports it; no arbitrary/unpinned package is installed. |

Narrow surface: no new code paths or network handlers are introduced. Publish integrity is
covered by the existing OIDC control + the human gate; no high-severity threats remain open.
</threat_model>
