<objective>
Close B1 (SEC-01, decision D-03): `PooledDatabase.execute` (and its async pendant) returns `cur.fetchall()` for `INSERT ... RETURNING` BEFORE `conn.commit()`, so the connection is returned to the pool with an open uncommitted transaction that rolls back — silent data loss. Fix so the commit happens before the rows are returned, sync and async. Add a dedicated red->green regression test.

Mark each as acquired (no new code) or, only if a genuine gap is found, close it. Produce a concrete acquired/gap checklist artifact.

Purpose: Prevent silent loss of `INSERT ... RETURNING` writes through the pool, and prove the v0.3.1 injection mitigations have not regressed.

  <action>
In `PooledDatabase.execute` (pool.py:142-158), the current control flow commits only in the no-`description` branch: when `cur.description` is truthy it `return cur.fetchall()` immediately, skipping `conn.commit()`, so an `INSERT ... RETURNING` returns its rows but the surrounding transaction is rolled back when the connection returns to the pool. Restructure so `conn.commit()` runs in BOTH branches before the function returns: capture `rows = cur.fetchall() if cur.description else []`, then `conn.commit()`, then `return rows`. The commit must execute while the connection is still checked out (inside the `with self.connection() as conn:` block).

Apply the identical fix to the async pendant `AsyncPooledDatabase.execute` (pool.py:341-357): `rows = await cur.fetchall() if cur.description else []`, then `await conn.commit()`, then `return rows`. Do NOT alter `execute_many` (already correct). Do NOT change method signatures or docstrings beyond what the fix requires.
  </action>
