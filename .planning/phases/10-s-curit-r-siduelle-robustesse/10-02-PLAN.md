<objective>
Close B3 (SEC-03, decision D-05): `Migrator._apply` (migrations.py:220-237) executes UP SQL then INSERT-version via a plain `with self.db.cursor()` block with no explicit transaction wrapping the two statements as a unit — a migration that fails after partially applying (e.g. multi-statement UP, or the INSERT failing after UP succeeded) can leave a partial schema and/or an inconsistent version table. The `rollback` path (migrations.py:286-297) has the same shape for DOWN SQL + DELETE-version. Wrap each pair in an EXPLICIT atomic transaction so a mid-course failure leaves NO partial trace. Add a dedicated red->green regression test.

Purpose: Guarantee all-or-nothing migration apply/rollback — protect schema integrity against partial application.
Output: Fixed `pycopg/migrations.py`, `tests/test_migration_atomicity.py`.
</objective>

In `Migrator._apply` (migrations.py:220-237), replace the bare `with self.db.cursor() as cur:` that runs `cur.execute(up_sql)` then the `INSERT INTO {self.table} (version, name) VALUES (%s, %s)` with an EXPLICIT atomic transaction so both statements commit together or roll back together. Use `self.db`'s transaction primitive: open `with self.db.transaction() as conn:` (database.py:323-341 — commits on success, rolls back on any exception) and run both statements through that connection's cursor (e.g. `with conn.cursor() as cur:`), keeping the existing parameterized INSERT exactly (do not change the SQL or the `%s` parameterization). On failure, psycopg rolls back the whole unit so neither the UP schema change nor the version row persists.

Apply the same restructure to the `rollback` per-migration block (migrations.py:286-297): run `cur.execute(down_sql)` and the parameterized `DELETE FROM {self.table} WHERE version = %s` inside one `with self.db.transaction() as conn:` so DOWN SQL + DELETE are atomic. Preserve the existing `try/except Exception -> raise MigrationError(...) from e` wrapping around the transaction so the error message and exception type are unchanged. Do NOT modify `migrate()` (the outer loop at 202-218), `_ensure_table`, `pending`, `applied`, or the silent-skip logging in `_get_migrations` (logging is locked out of scope).
