<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local repo → PyPI (public registry) | The publish step pushes an artifact to a public, immutable registry — the primary security-relevant boundary of this phase |
| local repo → GitHub (tag/release) | Tag + GitHub Release triggers the OIDC publish workflow |
| build artifact (wheel/sdist) → consumers | Anything bundled in the wheel ships to every installer — a secret leak here is public |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-06 | Tampering / Repudiation | `gh release create` → PyPI publish | mitigate | Publish is a BLOCKING human-verify checkpoint (Task 2) — no autonomous publish (RESEARCH Pitfall 6 / D-06). Irreversible step requires explicit human approval. |
| T-24-07 | Information disclosure | wheel/sdist contents | mitigate | Pre-publish wheel inspection (`unzip -l dist/pycopg-0.6.0-py3-none-any.whl`) confirms no `.env`, credentials, or `.planning/` content shipped — only the `pycopg/` package + metadata. |
| T-24-08 | Tampering | version confusion (wrong version published) | mitigate | `uv lock --check` + `python -c "import pycopg; assert pycopg.__version__ == '0.6.0'"` verified BEFORE tag/publish; both version sources grep-confirmed at 0.6.0. |
| T-24-SC | Tampering | publish credential / supply chain | mitigate | PyPI publish uses OIDC trusted publishing via `pypa/gh-action-pypi-publish` (publish.yml) — NO long-lived API token to leak. No new packages installed this phase (no `pip install` of third-party deps). |
</threat_model>
