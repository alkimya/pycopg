<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintainer → git tag/release | The `v0.4.0` tag and published GitHub Release are the irreversible trigger; must be created by the maintainer, not automation. |
| GitHub Actions → PyPI (OIDC) | `publish.yml` exchanges a short-lived OIDC token for PyPI upload rights — no long-lived secret crosses this boundary. |
| built artifact → public PyPI | The wheel/sdist become the permanent, immutable 0.4.0 distribution on PyPI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-06-SC | Tampering | npm/pip/cargo + action installs in publish pipeline | mitigate | publish.yml uses only official publishers: `astral-sh/setup-uv`, `actions/*`, and `pypa/gh-action-pypi-publish`. No new third-party install task is added in this plan. The node24 bumps (15-03) were verified against official releases pages. Task 1 checkpoint is the blocking human verification of the pipeline before the irreversible tag. |
| T-15-06-OIDC | Spoofing / Elevation of privilege | PyPI trusted publishing (OIDC) | mitigate | Confirm `id-token: write` + `pypa/gh-action-pypi-publish` are present and that NO long-lived PyPI API token or secret is hardcoded in any workflow (Task 1, step 3 grep). OIDC trusted publishing scopes upload rights to the `alkimya/pycopg` + `publish.yml` identity only — re-verified as configured (shipped v0.3.1 this way). |
| T-15-06-TAG | Tampering / Repudiation | git tag + GitHub Release trigger | mitigate | The tag and Release are maintainer-created via human-action checkpoints (Tasks 2–3); pushing `v0.4.0` is documented as the irreversible publish trigger. No automation creates the tag. |
| T-15-06-INT | Tampering | published wheel integrity | mitigate | The exact wheel was inspected in the Wave 2 gate (15-04) and re-confirmed in Task 1 (contains pycopg/spatial.py, version 0.4.0). publish.yml rebuilds from the same locked source (`uv lock --check`), so the published artifact matches the verified candidate. |
</threat_model>
