<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local/CI → PyPI (public index) | publishing an immutable artifact under the project's PyPI identity via OIDC; a wrong artifact cannot be unpublished/replaced at the same version |
| GitHub Release event → publish workflow | a Release-published event is the deliberate human trigger for the OIDC publish |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-02-01 | Elevation of privilege | unintended/automatic PyPI publish | mitigate | Publish is a `checkpoint:human-verify` gate="blocking-human" (Task 2) — never auto-advanced; the workflow only fires on a human-published GitHub Release |
| T-40-02-02 | Spoofing | publish credentials | mitigate | OIDC trusted publishing (no long-lived token); `id-token: write` + `pypi` environment already configured in publish.yml |
| T-40-02-03 | Tampering | wrong version shipped | mitigate | Task 1 tag points at the `version = "0.10.0"` commit; `uv lock --check` runs in the workflow; Task 3 clean-venv smoke asserts `__version__ == "0.10.0"` from the live PyPI artifact |
| T-40-02-SC | Tampering | clean-venv `pip install` from PyPI | mitigate | Installs the project's own just-published, pinned `pycopg==0.10.0` (no third-party [ASSUMED]/[SUS] packages); throwaway venv outside the tree |
</threat_model>
