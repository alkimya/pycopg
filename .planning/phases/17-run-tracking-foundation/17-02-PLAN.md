<objective>
Close the single verified Phase 17 gap (17-VERIFICATION.md, confirmed by 17-REVIEW.md CR-01):
the run-log write isolation is only CONDITIONAL today. `ETLAccessor.init` / `_start_run` /
`_end_run` call `self._db.execute(..., autocommit=True)`, which routes through the
session-aware `Database.cursor()`. When a `db.session()` is active, `cursor()` reuses the
session's transactional connection and SKIPS the commit block — so a `status='failed'`
run-log row written inside a session is rolled back together with the load it was supposed
to outlive. This breaks the D-04/D-05/ETL-08/ETL-09 invariant the phase exists to guarantee,
and it is invisible at the call site and untested.

This plan makes the isolation STRUCTURAL (Verifier option A): the three run-log write methods
open a dedicated connection directly via `self._db.connect(autocommit=True)` and run their SQL
on a cursor from THAT connection — bypassing `cursor()`'s session-reuse entirely. This satisfies
CONTEXT.md D-04 literally ("a fresh, short-lived autocommit connection per write via
db.connect(autocommit=True)"); the original implementation took the `db.execute()` shortcut,
which is where the conditional behavior crept in. A new regression test exercises the previously
untested session-active path and proves the invariant now holds there too.

Purpose: Phase 18 will build real load logic on this primitive and may use `db.session()` to
reuse one connection across extract + load (a documented, natural optimization). The isolation
must be safe under ANY consumer usage, not only the restricted no-session usage.

Output:
- `pycopg/etl.py`: `init` / `_start_run` / `_end_run` rewritten to use a dedicated
  `db.connect(autocommit=True)` + dict_row cursor (no `self._db.execute(...)` for run-log writes).
- `tests/test_etl_accessor.py`: `_FakeDatabase` updated to fake `connect()`+cursor (so the unit
  param-order tests survive the source change) + a new `test_failed_run_commits_inside_session`
  regression test.

Blast radius: `pycopg/etl.py` + `tests/test_etl_accessor.py` ONLY. `pycopg/database.py` is NOT
modified — the session/cursor behavior is left intact for every other library consumer
(`db.execute(autocommit=True)`, `pg_notify`, `create_database`, etc.). Verifier option B (changing
`Database.cursor()` itself) is rejected: it would change broad library semantics and risk
regressing the documented session-reuse contract. This is out of scope for a gap fix.

NOT in this plan: extract/transform/load logic (Phase 18/19), `RunResult`, `history()`,
`last_run()`, `AsyncETLAccessor`. The `run()` stub body is unchanged.
