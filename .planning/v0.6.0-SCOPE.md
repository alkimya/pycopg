# v0.6.0 — Scope de cadrage : Réorganisation en accessors

> **Statut : note de pré-cadrage (discussion).** Ce document capture les décisions prises en
> discussion avant le lancement de `/gsd-new-milestone`. Il n'est PAS un ROADMAP ni un
> REQUIREMENTS — il sert de base au questionnement de `/gsd-new-milestone`, qui produira les
> artefacts formels. Rédigé le 2026-06-15, après la clôture de v0.5.0.

## Thème

Assainir l'architecture de la classe `Database`/`AsyncDatabase` (monolithique, ~80 méthodes
publiques, ~2700 lignes) en regroupant les familles de méthodes sous des **accessors lazy**, sur
le modèle déjà éprouvé deux fois : `db.spatial.*` (v0.4.0) et `db.etl.*` (v0.5.0).

**Pourquoi maintenant (avant les features) :** principe « toolchain-first » validé en v0.4.0.
On réorganise la maison avant d'y ajouter des meubles. La fenêtre est idéale tant que la surface
reste gérable ; ajouter d'abord CRUD/TimescaleDB-avancé/spatial-v2 reviendrait à réorganiser une
surface ~30 % plus grosse, avec plus de dépréciations.

## Décisions VERROUILLÉES (prises en discussion — ne pas re-litiger)

- **D-SCOPE-1 — Stratégie de transition : alias + `DeprecationWarning`.**
  Chaque méthode déménagée garde son ancien nom à plat sur `db.*` / `async_db.*` comme **alias
  mince** qui (1) émet un `DeprecationWarning` pointant vers le nouveau chemin, (2) délègue à
  l'accessor. Suppression prévue en **v0.7.0**. Zéro rupture brutale — cohérent avec une lib
  publiée sur PyPI et avec la Core Value (« API propre »). NON retenu : breaking franc, double
  surface permanente.

- **D-SCOPE-2 — La vraie implémentation vit dans l'accessor ; l'ancien nom devient le wrapper.**
  `db.timescale.create_hypertable()` contient le code ; `db.create_hypertable()` est l'alias qui
  warn + délègue. Conséquence : supprimer la dette en v0.7.0 = effacer un bloc d'alias, sans
  toucher à la logique.

- **D-SCOPE-3 — Ampleur : les 5 accessors d'un seul milestone.**
  `db.timescale.*`, `db.admin.*`, `db.schema.*`, `db.maint.*`, `db.backup.*`. Le cœur
  transactionnel reste à plat sur `db.*`. Choix assumé malgré la taille (~5-6 phases) : le travail
  est mécanique et répétitif (déplacer + générer l'alias, le même geste ~50 fois), donc le pattern
  se valide à la phase 1 puis se réplique.

- **D-SCOPE-4 — Parité sync/async obligatoire (Core Value).** Chaque accessor a sa version async
  (`async_db.timescale.*`, etc.), et `test_parity` enregistre les nouveaux accessors comme il
  connaît déjà `db.spatial` / `db.etl`.

## Mapping méthode → accessor (surface réelle au 2026-06-15)

Source : `pycopg/database.py`. Toutes ont leur équivalent `async def` dans `async_database.py`
(parité vérifiée par échantillon : create_hypertable, create_role, grant, create_schema,
list_tables, vacuum, pg_dump → tous async=1).

### `db.timescale.*` (6 méthodes)
`create_hypertable`, `enable_compression`, `add_compression_policy`, `add_retention_policy`,
`list_hypertables`, `hypertable_info`
→ accueillera plus tard les helpers TSDB avancés (continuous aggregates, gapfill, chunks) — hors scope v0.6.0.

### `db.admin.*` (12 méthodes) — rôles & permissions
`create_role`, `drop_role`, `role_exists`, `list_roles`, `alter_role`, `grant_role`,
`revoke_role`, `grant`, `revoke`, `list_role_members`, `list_role_grants`
*(note : `grant`/`revoke` sont les permissions objets ; `grant_role`/`revoke_role` l'appartenance.)*

### `db.schema.*` (≈26 méthodes) — DDL, introspection, extensions, contraintes, index
Bases : `create_database`, `drop_database`, `database_exists`, `list_databases`
Extensions : `create_extension`, `drop_extension`, `list_extensions`, `has_extension`
Schémas : `create_schema`, `drop_schema`, `list_schemas`, `schema_exists`
Tables/colonnes : `list_tables`, `table_exists`, `list_columns`, `columns_with_types`,
`drop_table`, `truncate_table`, `table_info`, `row_count`
Contraintes/index : `add_primary_key`, `add_foreign_key`, `add_unique_constraint`,
`create_index`, `drop_index`, `list_indexes`, `list_constraints`
→ **C'est le plus gros bloc.** Question ouverte à trancher en cadrage : faut-il le sous-diviser
(`db.schema.*` pour DDL + `db.meta.*` pour introspection) ou le garder d'un bloc ? (voir Questions ouvertes)

### `db.maint.*` (6 méthodes) — maintenance & tailles
`size`, `table_size`, `table_sizes`, `vacuum`, `analyze`, `explain`

### `db.backup.*` (4 méthodes) — dump/restore & CSV
`pg_dump`, `pg_restore`, `copy_to_csv`, `copy_from_csv`

### RESTE À PLAT sur `db.*` (cœur transactionnel — NE PAS déménager)
`create`, `create_from_env`, `engine`, `connect`, `cursor`, `transaction`, `session`,
`in_session`, `execute`, `execute_many`, `insert_many`, `upsert_many`, `stream`, `notify`,
`insert_batch`, `copy_insert`, `fetch_one`, `fetch_val`, et les accessors existants
(`spatial`, `etl`).
*(à confirmer : `from_dataframe`/`to_dataframe`/`from_geodataframe`/`to_geodataframe` +
`create_spatial_index`/`list_geometry_columns` — DataFrame reste-t-il à plat ? le spatial-index
va-t-il sous `db.schema` ou `db.spatial` ? voir Questions ouvertes.)*

## Approche technique proposée (à valider en plan, non verrouillée)

- **Décorateur `@deprecated_alias("db.timescale.create_hypertable")`** plutôt que ~50 wrappers
  copier-collés : uniformise le `DeprecationWarning` (message + `stacklevel` correct), la
  délégation, et rend la suppression v0.7.0 triviale (un seul endroit à retirer). Même pattern
  côté async.
- **Accessors lazy** sur le modèle exact de `spatial`/`etl` : propriété `db.timescale` qui crée et
  cache une instance `TimescaleAccessor(self)` au premier accès. Builders SQL purs réutilisés tels
  quels (la logique ne change pas, elle déménage).
- **Exports `__init__.py`** : ajouter `TimescaleAccessor`/`AsyncTimescaleAccessor`,
  `AdminAccessor`/`Async…`, etc. dans `__all__`, sur le modèle `SpatialAccessor`/`ETLAccessor`
  déjà présents (lignes 51-55).

## Points de vigilance (pour le researcher/planner)

- **Ratchet de couverture (94 %)** : les alias ajoutent des lignes ; il faut que chaque alias soit
  couvert (au moins un test qui vérifie qu'il warn ET délègue correctement). Sinon la couverture
  baisse et le gate `--cov-fail-under=94` casse.
- **`test_parity`** : doit enregistrer les 5 nouveaux accessors. Risque que la parité casse si une
  méthode est déménagée côté sync mais pas async (ou inversement).
- **Filtrer le `DeprecationWarning` dans la suite de tests** : la suite appelle encore les anciens
  noms via les tests existants → soit migrer les tests vers les nouveaux chemins, soit
  `filterwarnings` ciblé, sinon le bruit de warnings explose (et un éventuel `-W error` casserait).
- **Cohérence des noms d'accessors** avec `spatial`/`etl` (snake_case, propriété lazy, même style
  de docstring numpydoc + `interrogate ≥ 95`).
- **Travail mécanique répétitif** : 5 accessors → découpage probable en ~1 phase par accessor
  (timescale/admin/maint/backup petits ; schema gros, peut-être 2 phases), + 1 phase de release.

## Questions ouvertes (à trancher pendant `/gsd-new-milestone` ou discuss-phase)

1. **`db.schema.*` est gros (~26 méthodes)** : le scinder en `db.schema.*` (DDL) + `db.meta.*`
   (introspection : `list_*`, `*_exists`, `table_info`, `columns_with_types`, `row_count`) ? Ou
   garder un seul accessor ?
2. **DataFrame** (`to_dataframe`/`from_dataframe`/`*_geodataframe`) : reste à plat sur `db.*`
   (usage très courant) ou sous un `db.frame.*` ? Penchant actuel : **à plat** (c'est du quotidien).
3. **`create_spatial_index` / `list_geometry_columns`** : restent où ? Candidats pour `db.spatial.*`
   (cohérence thématique) plutôt que `db.schema.*`.
4. **Numérotation de phase** : v0.5.0 s'est terminé à la Phase 20 → v0.6.0 commence à la **Phase 21**.
5. **Faut-il bumper la version mineure (0.6.0) ou est-ce un 0.5.x ?** Comme il y a dépréciation
   d'API publique (même non-breaking), **0.6.0** est justifié.

## Hors scope v0.6.0 (explicitement reporté)

- Tout **nouveau** helper (CRUD ergonomique, TimescaleDB avancé, spatial v2, introspection
  enrichie) — ce milestone **déménage l'existant**, il n'ajoute pas de pouvoir. Les nouveaux
  helpers viendront sur la surface propre dans des milestones ultérieurs.
- **ETL incrémental** (watermarks) — candidat séparé, indépendant de la réorganisation.
- **Suppression des alias** — planifiée pour v0.7.0, pas v0.6.0.
